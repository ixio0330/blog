[프런트엔드 개발을 위한 보안 입문](https://www.yes24.com/product/goods/124156282)이라는 책으로 진행하던 스터디가 끝나고, 네트워크(정확히는 HTTP) 스터디를 시작하게 되었다. 이번엔 책이 아닌 [SPDY 문서](https://www.chromium.org/spdy/spdy-whitepaper/), [RFC 문서](https://httpwg.org/specs/rfc7540.html) 두 개의 문서를 기반으로 진행하기로 했다.

이전까진 HTTP/1.1과 HTTP/2의 차이를 어렴풋이 알고만 있었는데 이번 기회에 확실하게 공부할 수 있어서 재미있었다.

## HTTP/1.1

HTTP/1.1은 HTTP/1.0에서 불가능하던 TCP 재사용이 가능해졌고, 응답을 기다리지 않고 연속적으로 요청을 보낼 수 있게 되었다. 하지만 웹 애플리케이션 복잡도가 올라가면서 HTTP/1.1의 한계가 점점 드러나기 시작했다.

1. HOL(Head-of-Line) Blocking
   - 연속적으로 요청을 보냈으나 응답은 순서대로 수신해야했다. A, B, C 세 개의 파일을 순서대로 요청했을 때, A라는 파일 크기가 크다면 B, C는 A를 모두 수신할 때 까지 기다릴 수 밖에 없다. 이를 우회하는 방법으로 브라우저는 2~6개의 TCP 연결을 사용했다.
2. 무거운 헤더
   - 매 요청마다 일반적으로 고정해서 사용하는 헤더를 중복 전송해야했다. 헤더의 데이터 크기가 커지면서 지연 시간이 발생했다.

## SPDY와 HTTP/2

HTTP/2는 구글의 SPDY 프로토콜을 기반으로 하고 있다. [SPDY 문서](https://www.chromium.org/spdy/spdy-whitepaper/)를 보면, SPDY는 `웹을 더 빠르게 만들자`는 프로젝트의 일환으로 지연 시간을 최소화하도록 설계했다고 한다.

위에서 언급한 두 가지 문제를 HTTP/2에서 어떻게 해결했을까?

### HOL Blocking을 멀티플렉싱

HTTP/2는 스트림이라는 가상의 통로를 통해 데이터를 프레임 단위로 전송한다. 하나의 연결 안에 여러 개의 스트림을 동시에 열 수 있고, 프레임의 스트림ID로 스트림을 구분한다. 이에 따라 A, B, C 세 개의 파일을 순서대로 요청한 상태에서 A라는 파일 크기가 커도 B, C를 동시에 수신할 수 있게 되었다.

애플리케이션 레벨의 HOL Blocking은 이렇게 해결했으나, TCP 자체의 패킷 손실로 인한 Transport레벨의 HOL Blocking은 존재한다. 스트림 중 하나라도 프레임이 유실되면 모든 통신이 중단되기 때문이다.

### HPACK 헤더 압축

HTTP/2는 정적/동적 테이블을 통해 자주 사용하는 헤더는 정적으로 번호를 정의해두고, 연결 중에 새롭게 사용한 헤더는 동적으로 저장한다. 그리고 이후엔 글자 대신 색인(index)만 전송하게 되는데, 이런 방식으로 HTTP 헤더를 압축하여 전송되는 패킷 수와 바이트 수를 줄였다.

만약 헤더 조립 중 하나라도 오류(`COMPRESSION_ERROR`)가 발생하면 보안과 무결성을 위해 연결을 즉시 종료한다.

### 또 하나의 장점, 서버 푸시

서버 푸시는 클라이언트가 요청하기 전에 서버가 먼저 리소스를 보내주는 기능이다. 예를 들어 index.html을 요청하면, 서버는 어차피 필요하게 될 style.css나 app.js를 기다리지 않고 미리 전송할 수 있다.

반면 이미 캐싱된 리소스도 무조건 푸시해버리는 경우가 있어, 불필요한 데이터를 전송하는 단점도 있다.

## 서버가 HTTP/2를 지원하는지 어떻게 알 수 있을까?

1. h2 (HTTPS): TLS 핸드쉐이크 과정에서 **ALPN(Application-Layer Protocol Negotiation)**을 통해 HTTP 버전을 결정한다.
2. h2c (HTTP): 평문 통신에서 사용하며, 먼저 HTTP/1.1로 연결을 한 후에 Upgrade 헤더를 통해 전환한다.

## 마치며

과거에 CSS 스프라이트 기법에 대해 배운적이 있다. 그 당시엔 '왜 아이콘을 하나의 파일로 구성해서 프론트엔드에서 한땀한땀 잘라서 써야하지?'라는 의문만 가진채 넘어갔었는데, HTTP/1.1의 한계로 인한 우회 방법이라는걸 알게 되었다.

HTTP/2가 표준 프로토콜이 되면서 반대로 파일을 더 작게 쪼개서 전송하는 것이 더 좋다.

아직 RFC 문서를 한번만 읽어봤는데 다시 읽어보고 더 꼼꼼하게 공부해서 스터디를 준비해야겠다.
