<a href="https://wig.dasoslab.com/" target="_blank">WIG</a>는 목표 설정부터 기록, 주간 점검까지 하나의 흐름으로 이어주는 목표 트래커다.

## 배경

독서모임에서 <a href="https://product.kyobobook.co.kr/detail/S000000597222?utm_source=google&utm_medium=cpc&utm_campaign=googleSearch&gad_source=1" target="_blank">성과를 내고 싶으면 실행하라</a>라는 책을 읽고 나서, 팀원들과 함께 책에서 제시한 방법대로 목표를 설정하고 목표를 달성할 수 있는 액션 아이템들을 설정해서 매주 일요일 밤 회고를 진행하고 있다. 작년 10월부터 시작했으니, 벌써 반년 정도 되었다.

책에서는 회고를 `가중목 회의`라고 하며, 가중목 회의에서는 `점수판`을 기반으로 잘한점, 보완할점, 다음주 계획 등을 발표한다. 점수판은 각자의 목표 달성 여부를 한눈에 확인할 수 있는 지표다. 각자 편한 방식으로 노션 / 구글 스프레드 시트 등을 이용해 점수판을 만들어서 사용해왔다.

나는 점수판을 노션으로 관리하고 있었는데 모바일 접근성이 안좋고, 항상 가중목 회의 직전에 점수판을 캡쳐해서 단톡에 공유하는게 귀찮았다. 사소한 불편함이 늘어나던 중 팀원중 한명이 가중목 시스템을 제품으로 만들어보라는 아이디어를 냈다.

## 앱 vs 웹

다만 요구사항이 있었는데, 앱으로 설치하고 푸시 알림이 오면 좋겠다는 의견이었다. 팀원들은 모두 아이폰을 사용하고 있었고, 나는 애플 개발자 계정이 유료가 아니기 때문에 앱을 원활하게 빌드하고 배포하기 어려워서 여러가지 대안을 생각해봤다.

1. 앱(Expo Go): 실제로 앱을 설치하는건 아니지만 Expo Go에서 앱을 실행시키는 꼼수를 생각해봤다. 하지만 원격 푸시 알림을 보내는데 제한이 있을 것으로 예상되어 패스!
2. 웹(PWA): 홈 화면에 추가할 수 있고, 원격 푸시 알림도 보낼 수 있다.

이 서비스는 네이티브 기능이 필요한 서비스가 아니기 때문에 MVP에선 웹으로 개발/배포하는게 더 편할 것 같다고 판단했다.

## 기획

기획을 할 때는 아마존의 6pages 문화 영향을 받아서 만든 `pre-result hook`을 기획 스킬로 만들어서 사용했다. 한창 클로드 코드로 스킬을 만들어서 개발할 때, 나 스스로를 LLM처럼 학습시킬 수 있지 않을까 하는 생각이 들었다. 그래서 생각을 할 때 6가지 질문을 해보자는 생각으로 만들었지만,, 실제로 습관으로 만드는건 어려웠다.

```md
pre-result hook: 어떤 제안, 생각, 아이디어가 생기면 결론을 내리기 전에 반드시 질문을 해본다.

- 어떤 배경, 맥락에서 발생했는가? or 어떤 문제가 있었는가?
- 왜 지금 해야하는가? or 다른 것이 아닌 이것을 왜 해야하는가?
- 이전과 이후가 어떻게 다른가? (나에게 혹은 고객에게)
- 실제로 어떤 작업을 해야하는가? (구체적인 액션 아이템 도출)
- 작업을 진행하면 시간과 비용은 얼마가 들 것으로 예상되는가?
- 이 제안, 생각, 아이디어가 어떻게 실패와 성공을 판단할 수 있는가?
```

그래도 스킬로 만들어서 LLM과 작업할 때 사용하니 생각을 구체화하는데 많은 도움이 되었다.

## 디자인

디자인은 별도로 하지 않았고, 리니어와 노션을 참고해서 LLM에게 브랜딩을 맡겼다. 브랜딩만 문서화 해두고 그 문서를 참조해서 프로토타입을 제작했다. 여러 버전으로 프로토타입을 제작하고, 마음에 드는 요소들을 골라 공통 컴포넌트화했다. 공통 컴포넌트로 만들지 않으면 매번 조금씩, 아주 미묘하게 다른 UI를 만들기 때문이다.

항상 느끼는건데 노베이스 상태에서는 클로드가 UI를 제일 잘 만든다. codex나 gemini는 이미지를 제공하면 동일하게 구현하는건 잘하지만, 아무것도 없는 상태에선 누가 봐도 AI가 만든 결과물을 뱉는다. AI니까 어쩔 수 없지만~

## 개발

프로토타입을 만들때는 클로드를 사용했지만, API나 API 연동 등의 로직을 개발할 때는 codex를 사용했다. 카카오톡 선물하기 대란때 하나 사둔 프로 구독권을 사용했고, 이 프로젝트에선 내가 코드를 단 한줄도 치지 않는걸 목표로 했다.

먼저 기획에 맞게 먼저 스키마를 설계하고, 스키마 기반으로 목 데이터를 생성해 프로토타입을 만들었다. 가중목 회의때 팀원들에게 프로토타입을 보여주면서 사용해볼 의향이 있는지 물어봤고, 다들 흔쾌히 사용하겠다고 해서 회의가 끝나자마자 개발하기 시작했다.

내가 직접 코드를 작성하진 않았지만, 스킬을 사용해 모든 피쳐를 동일한 프로세스로 개발했다.

- 백엔드: OpenAPI 스펙 정의 > 테스트 코드 작성 > API 개발
- 프론트엔드: Orval로 API 코드 생성 > UI 개발 > API 연동

밤 11시쯤 시작해서 새벽 3시쯤 동작하는 제품으로 만들고 QA 하면서 한시간 정도 고치니 동작하는 서비스를 완성했다. 이 제품을 5시간 만에 만들고 나서, 이제는 정말 누구나 아이디어를 검증하기 위한 프로토타입을 제작하고 실제 제품을 만드는게 가능해졌다는 사실이 실감났다.

## 마치며

WIG(Wildly Important Goals) 서비스명을 변경할까 고민중이다. 간편하게 가중목을 영어로 번역해서 줄인건데, wig 뜻이 가발이라서 모르는 사람이 보면 웃길 수 있을 것 같다는 생각이 계속 든다. ㅋㅋ ㅠ

그리고 이제 기능 개발보다는 마케팅에 집중해야 할 것 같다. 2주 정도 사용해보니 꽤 쓸만해서 실사용자를 확보해가면서 피드백 받고 이터레이션 하고싶다. 솔직히 브랜딩, 마케팅 이런 분야는 나한테 너무 어렵지만 부딪혀가면서 많은걸 배울 수 있을테니!
